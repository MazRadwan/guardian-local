## Schema Overview

Guardian's database schema has evolved through multiple epics:

**MVP (Epics 1-17):** Chat foundation + Questionnaire generation + Vendor tracking + Document upload
**Scoring (Epics 15, 18, 20):** Response extraction + Risk scoring + Narrative generation
**UX (Epic 25):** Chat title intelligence

### Core Tables (10 Tables)

```
users (system authentication)

vendors (who we assess)
  ↓
assessments (evaluation instances)
  ├── questions (dynamically generated by Claude)
  ├── responses (extracted vendor answers)
  ├── dimension_scores (per-dimension risk scores)
  └── assessment_results (composite scores + narratives)

conversations (chat sessions)
  ├── messages (chat history + attachments)
  └── files (uploaded documents + intake context)
```

**Current Capabilities:**
- ✅ Conversational AI chat (Consult / Assessment / Scoring modes)
- ✅ Generate customized assessment questionnaires
- ✅ Export questionnaires (PDF, Word, Excel)
- ✅ Track vendors and assessment history
- ✅ Multi-user support with basic auth
- ✅ Document upload with AI-powered intake parsing (Epic 16/17)
- ✅ Response extraction from uploaded questionnaires (Epic 15/18)
- ✅ AI-powered risk scoring across 10 dimensions (Epic 15/20)
- ✅ Narrative report generation (Epic 20)
- ✅ Auto-generated conversation titles (Epic 25)

**Future (Phase 3):**
- ⏸️ Portfolio analytics
- ⏸️ Comparative vendor analysis

### Design Philosophy

**MVP focuses on the conversation and questionnaire generation** - the core value Guardian provides. We validate the UX, then add complexity.

**Relational for:**
- Structured entities (vendors, users, assessments)
- Data needing queries (vendor history, user activity)

**JSONB for:**
- Flexible data (message content with components, vendor contact info)
- Data that varies by context (assessment metadata, conversation context)

**Why Drizzle:**
- SQL-first (full control over schema)
- Lightweight (~30KB runtime)
- No generation step (TypeScript inference)
- Team familiarity

### 1. users

**Purpose:** System users (analysts, admins). Authentication and authorization.

**Drizzle Schema:**
```typescript
import { pgTable, uuid, text, timestamp } from 'drizzle-orm/pg-core'

export const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),

  // Auth
  email: text('email').notNull().unique(),
  passwordHash: text('password_hash').notNull(),

  // Profile
  name: text('name').notNull(),
  role: text('role').notNull().$type<'admin' | 'analyst' | 'viewer'>().default('analyst'),

  // Session
  lastLoginAt: timestamp('last_login_at'),

  // Audit
  createdAt: timestamp('created_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
})
```

**Design Decisions:**
- **Simple 3 roles:** admin (full access), analyst (create assessments), viewer (read-only)
- **No soft delete for MVP:** Hard delete is fine, add `deactivatedAt` in Phase 2 if needed
- **No user metadata JSONB for MVP:** Keep it simple, add preferences later

**Feature Usage:**
- Login: `SELECT * FROM users WHERE email = $1`
- Authorization: Check role in JWT token (no DB query per request)
- User management: Admin can CRUD users

**Indexes:**
```typescript
// email is UNIQUE (automatic index)
```

### 3. assessments

**Purpose:** Assessment metadata and status. Links vendor to generated questionnaire and scoring results.

**Drizzle Schema:**
```typescript
export const assessments = pgTable('assessments', {
  id: uuid('id').primaryKey().defaultRandom(),
  vendorId: uuid('vendor_id').notNull().references(() => vendors.id, { onDelete: 'cascade' }),

  // Assessment details
  assessmentType: text('assessment_type').notNull().$type<'quick' | 'comprehensive' | 'category_focused'>(),
  solutionName: text('solution_name'), // "NLHS PMO Tool"
  solutionType: text('solution_type'), // "AI tool", "Cloud platform", etc.

  // Status (extended for scoring in Epic 15)
  status: text('status').notNull().$type<
    'draft' | 'questions_generated' | 'exported' | 'scored' | 'cancelled'
  >().default('draft'),

  // Metadata
  assessmentMetadata: jsonb('assessment_metadata').$type<{
    assessorName?: string
    stakeholders?: string[]
    notes?: string
  }>(),

  // Audit
  createdAt: timestamp('created_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
  createdBy: uuid('created_by').notNull().references(() => users.id),
})
```

**Design Decisions:**

**Assessment types:**
- `quick`: Rapid assessment with fewer questions
- `comprehensive`: Full 87+ question assessment
- `category_focused`: Assessment focused on specific risk dimensions

**Status workflow:**
- `draft`: Assessment created, questions not generated yet
- `questions_generated`: Questions ready for export
- `exported`: User downloaded questionnaire
- `scored`: Vendor responses scored (Epic 15)
- `cancelled`: User abandoned

**Feature Usage:**
- Create assessment: `INSERT INTO assessments (vendor_id, type, created_by) VALUES (...)` → When user starts assessment in chat
- Generate questions: `UPDATE assessments SET status = 'questions_generated' WHERE id = $1` → After Claude generates questions
- Track exports: `UPDATE assessments SET status = 'exported' WHERE id = $1` → When user downloads PDF/Word
- Mark scored: `UPDATE assessments SET status = 'scored' WHERE id = $1` → After scoring complete
- Vendor history: `SELECT * FROM assessments WHERE vendor_id = $1 ORDER BY created_at DESC` → Timeline

**Indexes:**
```typescript
export const assessmentsIndexes = {
  vendorIdx: index('assessments_vendor_idx').on(assessments.vendorId),
  statusIdx: index('assessments_status_idx').on(assessments.status),
  createdByIdx: index('assessments_created_by_idx').on(assessments.createdBy),
}
```

### 4. questions

**Purpose:** Stores Claude-generated assessment questions. Persisted for audit trail and questionnaire export.

**Drizzle Schema:**
```typescript
import { pgTable, uuid, text, integer, jsonb, timestamp, uniqueIndex } from 'drizzle-orm/pg-core'

export const questions = pgTable('questions', {
  id: uuid('id').primaryKey().defaultRandom(),
  assessmentId: uuid('assessment_id').notNull().references(() => assessments.id, { onDelete: 'cascade' }),

  // Organization
  sectionName: text('section_name').notNull(), // "Privacy Compliance", "Security Architecture"
  sectionNumber: integer('section_number').notNull(), // 1-11
  questionNumber: integer('question_number').notNull(), // Position within section

  // Content
  questionText: text('question_text').notNull(),
  questionType: text('question_type').notNull().$type<'text' | 'enum' | 'boolean'>().default('text'),

  questionMetadata: jsonb('question_metadata').$type<{
    required?: boolean
    helpText?: string
    enumOptions?: string[]
  }>(),

  createdAt: timestamp('created_at').defaultNow().notNull(),
}, (table) => ({
  uniquePosition: uniqueIndex('questions_assessment_position_idx')
    .on(table.assessmentId, table.sectionNumber, table.questionNumber),
}))
```

**Design Decisions:**
- **Why persist questions:** Immutable once generated (can't regenerate different questions mid-assessment)
- **Why section organization:** Questions grouped by risk dimension (Privacy, Security, etc.) for export formatting
- **Why unique constraint:** One question per position in assessment

**Feature Usage:**
- Generate questions: `INSERT INTO questions (assessment_id, section_name, question_text...) VALUES (...) × 87` → Bulk insert after Claude generates
- Export to PDF: `SELECT * FROM questions WHERE assessment_id = $1 ORDER BY section_number, question_number` → Render in document
- Display in chat: `SELECT * FROM questions WHERE assessment_id = $1 ORDER BY section_number, question_number` → For conversational mode (if we build it)

**Indexes:**
```typescript
export const questionsIndexes = {
  assessmentOrderIdx: index('questions_assessment_order_idx')
    .on(questions.assessmentId, questions.sectionNumber, questions.questionNumber),
}
```

### 5. conversations

**Purpose:** Chat sessions. Guardian is conversational, so conversations are core entities.

**Drizzle Schema:**
```typescript
import type { ConversationContext } from '../../../domain/entities/Conversation.js'

export const conversations = pgTable('conversations', {
  id: uuid('id').primaryKey().defaultRandom(),
  userId: uuid('user_id').notNull().references(() => users.id),

  // Mode (scoring added in Epic 15)
  mode: text('mode').notNull().$type<'consult' | 'assessment' | 'scoring'>().default('consult'),

  // Optional links
  assessmentId: uuid('assessment_id').references(() => assessments.id), // Nullable - consult mode has no assessment

  // Session state
  status: text('status').notNull().$type<'active' | 'completed'>().default('active'),
  // Domain type is source of truth for context structure (Epic 16: includes intakeContext)
  context: jsonb('context').$type<ConversationContext>(),

  // Epic 25: Chat Title Intelligence
  title: text('title'), // Generated or user-edited conversation title
  titleManuallyEdited: boolean('title_manually_edited').default(false).notNull(),

  // Timestamps
  startedAt: timestamp('started_at').defaultNow().notNull(),
  lastActivityAt: timestamp('last_activity_at').defaultNow().notNull(),
  completedAt: timestamp('completed_at'),
})
```

**Design Decisions:**
- **Three modes:** consult (general Q&A), assessment (questionnaire generation), scoring (risk analysis)
- **assessmentId nullable:** Consult conversations don't create assessments
- **Simple status:** active or completed (no paused state)
- **Title intelligence (Epic 25):** Auto-generated titles with manual edit protection

**Feature Usage:**
- Start conversation: `INSERT INTO conversations (user_id, mode) VALUES (...)` → User opens Guardian
- Switch mode: `UPDATE conversations SET mode = 'scoring' WHERE id = $1` → User selects mode from dropdown
- Link assessment: `UPDATE conversations SET assessment_id = $1 WHERE id = $1` → After assessment created
- Update title: `UPDATE conversations SET title = $1 WHERE id = $1 AND NOT title_manually_edited` → Auto-title
- Complete: `UPDATE conversations SET status = 'completed', completed_at = NOW() WHERE id = $1` → User closes chat

**Indexes:**
```typescript
export const conversationsIndexes = {
  userIdx: index('conversations_user_idx').on(conversations.userId),
  statusIdx: index('conversations_status_idx').on(conversations.status),
}
```

### 7. files (Epic 16/17)

**Purpose:** Uploaded documents with AI-extracted intake context. Supports multi-file upload per conversation.

**Drizzle Schema:**
```typescript
import { users } from './users'
import { conversations } from './conversations'

export const files = pgTable('files', {
  id: uuid('id').primaryKey().defaultRandom(),

  // Ownership
  userId: uuid('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
  conversationId: uuid('conversation_id').notNull().references(() => conversations.id, { onDelete: 'cascade' }),

  // File metadata
  filename: text('filename').notNull(),
  mimeType: text('mime_type').notNull(),
  size: integer('size').notNull(),
  storagePath: text('storage_path').notNull(),

  // Epic 17: Intake document context (parsed from uploaded files)
  intakeContext: jsonb('intake_context'),
  intakeGapCategories: text('intake_gap_categories').array(),
  intakeParsedAt: timestamp('intake_parsed_at', { withTimezone: true }),

  // Epic 18: Text excerpt for fast context injection
  textExcerpt: text('text_excerpt'),

  // Epic 18/31: Idempotency guard for parse/scoring operations
  parseStatus: varchar('parse_status', { length: 20 }).default('pending'), // 'pending' | 'complete' | 'failed'

  // Epic 18.4: Document type detection (heuristics)
  detectedDocType: varchar('detected_doc_type', { length: 20 }), // 'questionnaire' | 'document' | 'unknown'
  detectedVendorName: varchar('detected_vendor_name', { length: 255 }),

  // Timestamps
  createdAt: timestamp('created_at').defaultNow().notNull(),
}, (table) => ({
  conversationIdIdx: index('files_conversation_id_idx').on(table.conversationId),
}))
```

**IntakeDocumentContext Type:**
```typescript
interface IntakeDocumentContext {
  vendorName?: string
  solutionName?: string
  solutionType?: string
  features?: string[]
  securityClaims?: string[]
  complianceClaims?: string[]
  integrations?: string[]
  dataHandling?: string[]
  aiCapabilities?: string[]
}
```

**Design Decisions:**
- **Per-file context storage:** Each file stores its own parsed intake context (Epic 17 - concurrency safe)
- **Aggregation at query time:** `findByConversationWithContext()` retrieves all files, ChatServer aggregates
- **storagePath never exposed:** Clients use `fileId` (database UUID), server resolves storage path internally
- **Gap categories:** AI identifies missing risk dimensions for follow-up questions

**Feature Usage:**
- Upload file: `INSERT INTO files (user_id, conversation_id, filename, ...) VALUES (...)`
- Store parsed context: `UPDATE files SET intake_context = $1, intake_parsed_at = NOW() WHERE id = $2`
- Get all context for conversation: `SELECT * FROM files WHERE conversation_id = $1 AND intake_context IS NOT NULL ORDER BY intake_parsed_at, id`
- Download file: Lookup by `id`, resolve `storagePath`, stream from storage

**Indexes:**
```typescript
export const filesIndexes = {
  conversationIdIdx: index('files_conversation_id_idx').on(files.conversationId),
}
```

**Security:**
- Files validated by magic bytes, not just extension (FileValidationService)
- Storage paths never sent to client
- Conversation ownership verified before access

### 8. responses

**Purpose:** Store extracted vendor answers from uploaded questionnaires. Supports AI-powered extraction with confidence scores.

**Drizzle Schema:**
```typescript
export const responses = pgTable('responses', {
  id: uuid('id').primaryKey().defaultRandom(),
  assessmentId: uuid('assessment_id').notNull().references(() => assessments.id, { onDelete: 'cascade' }),
  batchId: uuid('batch_id').notNull(), // Groups responses from single upload
  fileId: uuid('file_id').references(() => files.id), // Optional link to source file

  // Question identification (by position, not FK)
  sectionNumber: integer('section_number').notNull(),
  questionNumber: integer('question_number').notNull(),

  // Extracted content
  questionText: text('question_text').notNull(),
  responseText: text('response_text').notNull(),
  confidence: real('confidence'), // Extraction confidence 0-1
  hasVisualContent: boolean('has_visual_content').default(false),
  visualContentDescription: text('visual_content_description'),

  createdAt: timestamp('created_at').defaultNow().notNull(),
}, (table) => ({
  assessmentBatchIdx: index('responses_assessment_batch_idx').on(table.assessmentId, table.batchId),
  positionIdx: index('responses_position_idx').on(table.assessmentId, table.sectionNumber, table.questionNumber),
  batchIdIdx: index('responses_batch_id_idx').on(table.batchId),
  createdAtIdx: index('responses_created_at_idx').on(table.createdAt),
}))
```

**Design Decisions:**
- **Batch-based extraction:** Each upload creates a batch of responses
- **Position-based matching:** Matches by section/question number, not FK (handles questionnaire variations)
- **Confidence scoring:** AI provides extraction confidence for QA
- **Visual content support:** Tracks diagrams/charts that can't be fully extracted as text

**Feature Usage:**
- Extract responses: `INSERT INTO responses (assessment_id, batch_id, ...) VALUES (...)` → Bulk insert after extraction
- Get responses for scoring: `SELECT * FROM responses WHERE assessment_id = $1 ORDER BY section_number, question_number`
- Orphan cleanup: `DELETE FROM responses WHERE batch_id NOT IN (SELECT batch_id FROM assessment_results) AND created_at < $1`

### 9. dimension_scores

**Purpose:** Per-dimension risk scores with detailed findings. One row per dimension per scoring batch.

**Drizzle Schema:**
```typescript
export const dimensionScores = pgTable('dimension_scores', {
  id: uuid('id').primaryKey().defaultRandom(),
  assessmentId: uuid('assessment_id').notNull().references(() => assessments.id, { onDelete: 'cascade' }),
  batchId: uuid('batch_id').notNull(),

  dimension: text('dimension').notNull(), // 'clinical_risk', 'privacy_risk', etc.
  score: integer('score').notNull(), // 0-100
  riskRating: text('risk_rating').notNull().$type<'low' | 'medium' | 'high' | 'critical'>(),

  // Detailed findings
  findings: jsonb('findings').$type<{
    subScores: Array<{ name: string; score: number; maxScore: number; notes: string }>
    keyRisks: string[]
    mitigations: string[]
    evidenceRefs: Array<{ sectionNumber: number; questionNumber: number; quote: string }>
  }>(),

  createdAt: timestamp('created_at').defaultNow().notNull(),
}, (table) => ({
  assessmentDimensionIdx: index('dimension_scores_assessment_idx').on(table.assessmentId, table.dimension),
  uniqueBatchDimension: unique('dimension_scores_batch_dimension_unique').on(
    table.assessmentId, table.batchId, table.dimension
  ),
}))
```

**10 Risk Dimensions:**
1. `clinical_risk` - Patient safety and clinical workflow
2. `privacy_risk` - Data privacy and HIPAA compliance
3. `security_risk` - Cybersecurity and access control
4. `operational_risk` - System reliability and uptime
5. `integration_risk` - EHR/system integration complexity
6. `vendor_risk` - Vendor stability and support
7. `regulatory_risk` - Compliance with healthcare regulations
8. `financial_risk` - Cost and ROI considerations
9. `change_management_risk` - Implementation and training
10. `data_governance_risk` - Data quality and lifecycle

**Feature Usage:**
- Store dimension score: `INSERT INTO dimension_scores (assessment_id, batch_id, dimension, ...) VALUES (...)`
- Get all scores: `SELECT * FROM dimension_scores WHERE assessment_id = $1 AND batch_id = $2`
- Idempotency: Unique constraint prevents duplicate scores per dimension per batch

### 10. assessment_results

**Purpose:** Composite scoring results with narrative reports. One row per scoring batch.

**Drizzle Schema:**
```typescript
export type NarrativeStatus = 'generating' | 'complete' | 'failed'

export const assessmentResults = pgTable('assessment_results', {
  id: uuid('id').primaryKey().defaultRandom(),
  assessmentId: uuid('assessment_id').notNull().references(() => assessments.id, { onDelete: 'cascade' }),
  batchId: uuid('batch_id').notNull(),

  // Composite scoring
  compositeScore: integer('composite_score').notNull(), // 0-100
  recommendation: text('recommendation').notNull().$type<'approve' | 'conditional' | 'decline' | 'more_info'>(),
  overallRiskRating: text('overall_risk_rating').notNull().$type<'low' | 'medium' | 'high' | 'critical'>(),

  // Generated content
  narrativeReport: text('narrative_report'), // Full markdown from Claude
  executiveSummary: text('executive_summary'),
  keyFindings: jsonb('key_findings').$type<string[]>(),
  disqualifyingFactors: jsonb('disqualifying_factors').$type<string[]>(),

  // Narrative generation status (Epic 20: concurrency-safe claim pattern)
  narrativeStatus: varchar('narrative_status', { length: 20 }).$type<NarrativeStatus>(),
  narrativeClaimedAt: timestamp('narrative_claimed_at', { withTimezone: true }),
  narrativeCompletedAt: timestamp('narrative_completed_at', { withTimezone: true }),
  narrativeError: text('narrative_error'),

  // Provenance (for auditability)
  rubricVersion: text('rubric_version').notNull(), // 'guardian-v1.0'
  modelId: text('model_id').notNull(), // 'claude-sonnet-4-5-20250929'
  rawToolPayload: jsonb('raw_tool_payload'), // Original scoring_complete payload

  // Audit
  scoredAt: timestamp('scored_at').defaultNow().notNull(),
  scoringDurationMs: integer('scoring_duration_ms'),
}, (table) => ({
  assessmentIdx: index('assessment_results_assessment_idx').on(table.assessmentId),
  uniqueBatch: unique('assessment_results_batch_unique').on(table.assessmentId, table.batchId),
  narrativeStatusIdx: index('assessment_results_narrative_status_idx').on(table.narrativeStatus),
}))
```

**Design Decisions:**
- **Batch idempotency:** One result per scoring batch prevents duplicates
- **Claim pattern (Epic 20):** Concurrency-safe narrative generation with status tracking
- **Provenance tracking:** Stores rubric version and model ID for audit trail
- **Raw payload storage:** Preserves original Claude response for debugging

**Recommendation values:**
- `approve`: Low risk, proceed with vendor
- `conditional`: Medium risk, proceed with mitigations
- `decline`: High risk, do not proceed
- `more_info`: Insufficient information to score

**Feature Usage:**
- Store result: `INSERT INTO assessment_results (assessment_id, batch_id, ...) VALUES (...)`
- Claim narrative: `UPDATE assessment_results SET narrative_status = 'generating', narrative_claimed_at = NOW() WHERE id = $1 AND narrative_status IS NULL`
- Complete narrative: `UPDATE assessment_results SET narrative_status = 'complete', narrative_report = $1 WHERE id = $2`

## Relationships

```
users
  ├── (created_by) → assessments
  ├── (user_id) → conversations
  └── (user_id) → files

vendors
  └── (vendor_id) → assessments
        ├── (assessment_id) → questions
        ├── (assessment_id) → responses
        ├── (assessment_id) → dimension_scores
        └── (assessment_id) → assessment_results

conversations
  ├── (user_id) → users
  ├── (assessment_id) → assessments (nullable)
  ├── (conversation_id) → messages
  └── (conversation_id) → files

files
  └── (file_id) → responses (optional source link)
```

**Foreign Key Cascade Rules:**

| Parent | Child | ON DELETE | Rationale |
|--------|-------|-----------|-----------|
| vendors | assessments | CASCADE | If vendor deleted, assessments go with it |
| assessments | questions | CASCADE | Questions belong to assessment |
| assessments | responses | CASCADE | Responses belong to assessment |
| assessments | dimension_scores | CASCADE | Scores belong to assessment |
| assessments | assessment_results | CASCADE | Results belong to assessment |
| conversations | messages | CASCADE | Messages belong to conversation |
| conversations | files | CASCADE | Files belong to conversation |
| users | assessments | RESTRICT | Can't delete user who created assessments |
| users | conversations | RESTRICT | Can't delete user who has conversations |
| users | files | CASCADE | If user deleted, their files go with them |
| files | responses | SET NULL | Responses can exist without source file |

## Migration Strategy

### Initial Migration

```sql
-- migrations/0001_create_mvp_tables.sql

-- Users table
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  name TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'analyst',
  last_login_at TIMESTAMP,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Vendors table
CREATE TABLE vendors (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  industry TEXT,
  website TEXT,
  contact_info JSONB,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX vendors_name_idx ON vendors(name);

-- Assessments table
CREATE TABLE assessments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vendor_id UUID NOT NULL REFERENCES vendors(id) ON DELETE CASCADE,
  assessment_type TEXT NOT NULL,
  solution_name TEXT,
  solution_type TEXT,
  status TEXT NOT NULL DEFAULT 'draft',
  assessment_metadata JSONB,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
  created_by UUID NOT NULL REFERENCES users(id)
);

CREATE INDEX assessments_vendor_idx ON assessments(vendor_id);
CREATE INDEX assessments_status_idx ON assessments(status);
CREATE INDEX assessments_created_by_idx ON assessments(created_by);

-- Questions table
CREATE TABLE questions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  assessment_id UUID NOT NULL REFERENCES assessments(id) ON DELETE CASCADE,
  section_name TEXT NOT NULL,
  section_number INTEGER NOT NULL,
  question_number INTEGER NOT NULL,
  question_text TEXT NOT NULL,
  question_type TEXT NOT NULL DEFAULT 'text',
  question_metadata JSONB,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  UNIQUE(assessment_id, section_number, question_number)
);

CREATE INDEX questions_assessment_order_idx ON questions(assessment_id, section_number, question_number);

-- Conversations table
CREATE TABLE conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id),
  mode TEXT NOT NULL DEFAULT 'consult',
  assessment_id UUID REFERENCES assessments(id),
  status TEXT NOT NULL DEFAULT 'active',
  context JSONB,
  started_at TIMESTAMP NOT NULL DEFAULT NOW(),
  last_activity_at TIMESTAMP NOT NULL DEFAULT NOW(),
  completed_at TIMESTAMP
);

CREATE INDEX conversations_user_idx ON conversations(user_id);
CREATE INDEX conversations_status_idx ON conversations(status);

-- Messages table
CREATE TABLE messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  role TEXT NOT NULL,
  content JSONB NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX messages_conversation_idx ON messages(conversation_id);
CREATE INDEX messages_created_at_idx ON messages(created_at);
```

## Feature Map

### What Each Table Supports

**Chat Foundation:**
- `users`: Who's chatting
- `conversations`: Chat sessions with mode switching and auto-titles
- `messages`: Chat history, context for resuming

**Questionnaire Generation:**
- `vendors`: Who we're assessing
- `assessments`: Track assessment lifecycle from draft to scored
- `questions`: The 87 questions Claude generated for this vendor
- `files`: Uploaded documents for intake parsing

**Risk Scoring (Epic 15/18/20):**
- `responses`: Extracted vendor answers from uploaded questionnaires
- `dimension_scores`: Per-dimension risk scores (10 dimensions)
- `assessment_results`: Composite scores, recommendations, and narrative reports

**What's NOT Yet Implemented:**
- ❌ Detailed audit log (planned for Phase 3)
- ❌ Portfolio analytics

## Performance Considerations

**Queries that need to be fast:**
1. ✅ Load conversation history (last 50 messages) - Indexed on conversation_id
2. ✅ Get vendor's assessments - Indexed on vendor_id
3. ✅ Load questions for assessment - Indexed on assessment_id
4. ✅ User's active conversations - Indexed on user_id
5. ✅ Get responses for scoring - Indexed on assessment_id + batch_id
6. ✅ Get dimension scores - Indexed on assessment_id + dimension
7. ✅ Claim narrative generation - Indexed on narrative_status

**All queries hit indexes.** No optimization needed at current scale.
