## Schema Overview

Guardian's database schema has evolved through multiple epics:

**MVP (Epics 1-17):** Chat foundation + Questionnaire generation + Vendor tracking + Document upload
**Scoring (Epics 15, 18, 20, 40):** Response extraction + Risk scoring (Rubric v1.1, 10-dimension weighted) + Narrative generation
**UX (Epic 25):** Chat title intelligence
**ISO Compliance (Epic 37):** Compliance framework reference data + Assessment compliance results

### Table Groups (16 Tables)

```
users (system authentication)

vendors (who we assess)
  |
assessments (evaluation instances)
  |-- questions (dynamically generated by Claude)
  |-- responses (extracted vendor answers)
  |-- dimension_scores (per-dimension risk scores with sub-scores)
  |-- assessment_results (composite scores + narratives, Rubric v1.1)
  +-- assessment_compliance_results (per-control ISO findings)

conversations (chat sessions)
  |-- messages (chat history + attachments)
  +-- files (uploaded documents + intake context)

compliance_frameworks (ISO standard definitions)
  +-- framework_versions (versioned releases)
        +-- framework_controls (individual clauses/controls)
              |-- interpretive_criteria (Guardian's assessment language)
              +-- dimension_control_mappings (link controls to risk dimensions)
```

**Current Capabilities:**
- Conversational AI chat (Consult / Assessment / Scoring modes)
- Generate customized assessment questionnaires
- Export questionnaires (PDF, Word, Excel)
- Track vendors and assessment history
- Multi-user support with basic auth
- Document upload with AI-powered intake parsing (Epic 16/17)
- Response extraction from uploaded questionnaires (Epic 15/18)
- AI-powered risk scoring across 10 weighted dimensions, Rubric v1.1 (Epic 15/20/40)
- ScoringPayloadReconciler auto-corrects Claude's arithmetic (Epic 40)
- Narrative report generation (Epic 20)
- Auto-generated conversation titles (Epic 25)
- ISO compliance framework reference data (Epic 37)
- Per-assessment ISO compliance results (Epic 37)

**Future (Phase 3):**
- Portfolio analytics
- Comparative vendor analysis

### Design Philosophy

**MVP focuses on the conversation and questionnaire generation** - the core value Guardian provides. We validate the UX, then add complexity.

**Relational for:**
- Structured entities (vendors, users, assessments)
- Data needing queries (vendor history, user activity)
- Compliance framework hierarchy (frameworks > versions > controls)

**JSONB for:**
- Flexible data (message content with components, vendor contact info)
- Data that varies by context (assessment metadata, conversation context)
- Detailed findings (sub-scores, evidence references, ISO clause references)

**Why Drizzle:**
- SQL-first (full control over schema)
- Lightweight (~30KB runtime)
- No generation step (TypeScript inference)
- Team familiarity

### 1. users

**Purpose:** System users (analysts, admins). Authentication and authorization.

**Drizzle Schema:**
```typescript
import { pgTable, uuid, text, timestamp } from 'drizzle-orm/pg-core'

export const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),

  // Auth
  email: text('email').notNull().unique(),
  passwordHash: text('password_hash').notNull(),

  // Profile
  name: text('name').notNull(),
  role: text('role').notNull().$type<'admin' | 'analyst' | 'viewer'>().default('analyst'),

  // Session
  lastLoginAt: timestamp('last_login_at'),

  // Audit
  createdAt: timestamp('created_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
```

**Design Decisions:**
- **Simple 3 roles:** admin (full access), analyst (create assessments), viewer (read-only)
- **No soft delete for MVP:** Hard delete is fine, add `deactivatedAt` in Phase 2 if needed
- **No user metadata JSONB for MVP:** Keep it simple, add preferences later

**Feature Usage:**
- Login: `SELECT * FROM users WHERE email = $1`
- Authorization: Check role in JWT token (no DB query per request)
- User management: Admin can CRUD users

### 3. assessments

**Purpose:** Assessment metadata and status. Links vendor to generated questionnaire and scoring results.

**Drizzle Schema:**
```typescript
export const assessments = pgTable(
  'assessments',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    vendorId: uuid('vendor_id').notNull().references(() => vendors.id, { onDelete: 'cascade' }),

    // Assessment details
    assessmentType: text('assessment_type').notNull()
      .$type<'quick' | 'comprehensive' | 'category_focused'>(),
    solutionName: text('solution_name'),
    solutionType: text('solution_type'),

    // Status (extended for scoring in Epic 15)
    status: text('status').notNull()
      .$type<'draft' | 'questions_generated' | 'exported' | 'scored' | 'cancelled'>()
      .default('draft'),

    // Metadata
    assessmentMetadata: jsonb('assessment_metadata').$type<{
      assessorName?: string
      stakeholders?: string[]
      notes?: string
    }>(),

    // Audit
    createdAt: timestamp('created_at').defaultNow().notNull(),
    updatedAt: timestamp('updated_at').defaultNow().notNull(),
    createdBy: uuid('created_by').notNull().references(() => users.id),
  },
  (table) => ({
    vendorIdx: index('assessments_vendor_idx').on(table.vendorId),
    statusIdx: index('assessments_status_idx').on(table.status),
    createdByIdx: index('assessments_created_by_idx').on(table.createdBy),
  })
)
```

**Design Decisions:**

**Assessment types:**
- `quick`: Rapid assessment with fewer questions
- `comprehensive`: Full 87+ question assessment
- `category_focused`: Assessment focused on specific risk dimensions

**Status workflow:**
- `draft`: Assessment created, questions not generated yet
- `questions_generated`: Questions ready for export
- `exported`: User downloaded questionnaire
- `scored`: Vendor responses scored (Epic 15)
- `cancelled`: User abandoned

**Feature Usage:**
- Create assessment: `INSERT INTO assessments (vendor_id, type, created_by) VALUES (...)` -- When user starts assessment in chat
- Generate questions: `UPDATE assessments SET status = 'questions_generated' WHERE id = $1` -- After Claude generates questions
- Track exports: `UPDATE assessments SET status = 'exported' WHERE id = $1` -- When user downloads PDF/Word
- Mark scored: `UPDATE assessments SET status = 'scored' WHERE id = $1` -- After scoring complete
- Vendor history: `SELECT * FROM assessments WHERE vendor_id = $1 ORDER BY created_at DESC` -- Timeline

**Indexes:**
```typescript
vendorIdx: index('assessments_vendor_idx').on(table.vendorId)
statusIdx: index('assessments_status_idx').on(table.status)
createdByIdx: index('assessments_created_by_idx').on(table.createdBy)
```

```typescript
    detectedDocType: varchar('detected_doc_type', { length: 20 }), // 'questionnaire' | 'document' | 'unknown'

```

**IntakeDocumentContext Type:**
```typescript
interface IntakeDocumentContext {
  vendorName?: string
  solutionName?: string
  solutionType?: string
  features?: string[]
  securityClaims?: string[]
  complianceClaims?: string[]
  integrations?: string[]
  dataHandling?: string[]
  aiCapabilities?: string[]
}
```

**Design Decisions:**
- **Per-file context storage:** Each file stores its own parsed intake context (Epic 17 - concurrency safe)
- **Aggregation at query time:** `findByConversationWithContext()` retrieves all files, ChatServer aggregates
- **storagePath never exposed:** Clients use `fileId` (database UUID), server resolves storage path internally
- **Gap categories:** AI identifies missing risk dimensions for follow-up questions

**Feature Usage:**
- Upload file: `INSERT INTO files (user_id, conversation_id, filename, ...) VALUES (...)`
- Store parsed context: `UPDATE files SET intake_context = $1, intake_parsed_at = NOW() WHERE id = $2`
- Get all context for conversation: `SELECT * FROM files WHERE conversation_id = $1 AND intake_context IS NOT NULL ORDER BY intake_parsed_at, id`
- Download file: Lookup by `id`, resolve `storagePath`, stream from storage

**Indexes:**
```typescript
conversationIdIdx: index('files_conversation_id_idx').on(table.conversationId)
```

**Security:**
- Files validated by magic bytes, not just extension (FileValidationService)
- Storage paths never sent to client
- Conversation ownership verified before access

### 9. dimension_scores

**Purpose:** Per-dimension risk scores with detailed findings. One row per dimension per scoring batch.

**Drizzle Schema:**
```typescript
export const dimensionScores = pgTable(
  'dimension_scores',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    assessmentId: uuid('assessment_id').notNull().references(() => assessments.id, { onDelete: 'cascade' }),
    batchId: uuid('batch_id').notNull(),

    dimension: text('dimension').notNull(), // 'clinical_risk', 'privacy_risk', etc.
    score: integer('score').notNull(), // 0-100
    riskRating: text('risk_rating').notNull().$type<'low' | 'medium' | 'high' | 'critical'>(),

    // Detailed findings
    findings: jsonb('findings').$type<{
      subScores: Array<{ name: string; score: number; maxScore: number; notes: string }>
      keyRisks: string[]
      mitigations: string[]
      evidenceRefs: Array<{ sectionNumber: number; questionNumber: number; quote: string }>
      // ISO enrichment (Epic 37)
      assessmentConfidence?: { level: 'high' | 'medium' | 'low'; rationale: string }
      isoClauseReferences?: Array<{
        clauseRef: string
        title: string
        framework: string
        status: 'aligned' | 'partial' | 'not_evidenced' | 'not_applicable'
      }>
    }>(),

    createdAt: timestamp('created_at').defaultNow().notNull(),
  },
  (table) => ({
    assessmentDimensionIdx: index('dimension_scores_assessment_idx').on(table.assessmentId, table.dimension),
    uniqueBatchDimension: unique('dimension_scores_batch_dimension_unique').on(
      table.assessmentId, table.batchId, table.dimension
    ),
  })
)
```

**10 Risk Dimensions:**
1. `clinical_risk` - Patient safety and clinical workflow
2. `privacy_risk` - Data privacy and HIPAA compliance
3. `security_risk` - Cybersecurity and access control
4. `operational_risk` - System reliability and uptime
5. `integration_risk` - EHR/system integration complexity
6. `vendor_risk` - Vendor stability and support
7. `regulatory_risk` - Compliance with healthcare regulations
8. `financial_risk` - Cost and ROI considerations
9. `change_management_risk` - Implementation and training
10. `data_governance_risk` - Data quality and lifecycle

**Findings JSONB structure (Rubric v1.1):**
- `subScores`: Each dimension has sub-score rules; the dimension score equals the sum of sub-scores. The ScoringPayloadReconciler auto-corrects if Claude's arithmetic drifts.
- `keyRisks`: Critical risk factors identified in vendor responses
- `mitigations`: Vendor-stated or recommended mitigations
- `evidenceRefs`: Links back to specific questionnaire Q&A pairs
- `assessmentConfidence` (optional, Epic 37): AI confidence in the assessment for this dimension
- `isoClauseReferences` (optional, Epic 37): Relevant ISO clause references with alignment status

**Feature Usage:**
- Store dimension score: `INSERT INTO dimension_scores (assessment_id, batch_id, dimension, ...) VALUES (...)`
- Get all scores: `SELECT * FROM dimension_scores WHERE assessment_id = $1 AND batch_id = $2`
- Idempotency: Unique constraint prevents duplicate scores per dimension per batch

### 10. assessment_results

**Purpose:** Composite scoring results with narrative reports. One row per scoring batch.

**Drizzle Schema:**
```typescript
export type NarrativeStatus = 'generating' | 'complete' | 'failed'

export const assessmentResults = pgTable(
  'assessment_results',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    assessmentId: uuid('assessment_id').notNull().references(() => assessments.id, { onDelete: 'cascade' }),
    batchId: uuid('batch_id').notNull(),

    // Composite scoring
    compositeScore: integer('composite_score').notNull(), // 0-100
    recommendation: text('recommendation').notNull()
      .$type<'approve' | 'conditional' | 'decline' | 'more_info'>(),
    overallRiskRating: text('overall_risk_rating').notNull()
      .$type<'low' | 'medium' | 'high' | 'critical'>(),

    // Generated content
    narrativeReport: text('narrative_report'), // Full markdown from Claude
    executiveSummary: text('executive_summary'),
    keyFindings: jsonb('key_findings').$type<string[]>(),
    disqualifyingFactors: jsonb('disqualifying_factors').$type<string[]>(),

    // Narrative generation status (Epic 20: concurrency-safe claim pattern)
    narrativeStatus: varchar('narrative_status', { length: 20 }).$type<NarrativeStatus>(),
    narrativeClaimedAt: timestamp('narrative_claimed_at', { withTimezone: true }),
    narrativeCompletedAt: timestamp('narrative_completed_at', { withTimezone: true }),
    narrativeError: text('narrative_error'),

    // Provenance (for auditability)
    rubricVersion: text('rubric_version').notNull(), // 'guardian-v1.1'
    modelId: text('model_id').notNull(), // 'claude-sonnet-4-5-20250929'
    rawToolPayload: jsonb('raw_tool_payload'), // Original scoring_complete payload

    // Audit
    scoredAt: timestamp('scored_at').defaultNow().notNull(),
    scoringDurationMs: integer('scoring_duration_ms'),
  },
  (table) => ({
    assessmentIdx: index('assessment_results_assessment_idx').on(table.assessmentId),
    uniqueBatch: unique('assessment_results_batch_unique').on(table.assessmentId, table.batchId),
    narrativeStatusIdx: index('assessment_results_narrative_status_idx').on(table.narrativeStatus),
  })
)
```

**Design Decisions:**
- **Batch idempotency:** One result per scoring batch prevents duplicates
- **Claim pattern (Epic 20):** Concurrency-safe narrative generation with status tracking
- **Provenance tracking:** Stores rubric version and model ID for audit trail
- **Raw payload storage:** Preserves original Claude response for debugging
- **ScoringPayloadReconciler (Epic 40):** Before persisting, the reconciler auto-corrects dimension scores (from sub-score sums), composite scores (from weighted average), and recommendation coherence (from disqualifiers). Deviations are logged as warnings, not structural violations.

**Recommendation values:**
- `approve`: Low risk, proceed with vendor
- `conditional`: Medium risk, proceed with mitigations
- `decline`: High risk, do not proceed
- `more_info`: Insufficient information to score

**Feature Usage:**
- Store result: `INSERT INTO assessment_results (assessment_id, batch_id, ...) VALUES (...)`
- Claim narrative: `UPDATE assessment_results SET narrative_status = 'generating', narrative_claimed_at = NOW() WHERE id = $1 AND narrative_status IS NULL`
- Complete narrative: `UPDATE assessment_results SET narrative_status = 'complete', narrative_report = $1 WHERE id = $2`

## ISO Compliance Tables (Epic 37)

These tables store ISO compliance framework reference data and per-assessment compliance evaluation results. The hierarchy is: frameworks > versions > controls > interpretive criteria, with a many-to-many mapping between controls and Guardian's 10 risk dimensions.

### 11. compliance_frameworks

**Purpose:** Top-level ISO/IEC standard definitions (e.g., ISO/IEC 42001, ISO/IEC 23894).

**Drizzle Schema:**
```typescript
export const complianceFrameworks = pgTable(
  'compliance_frameworks',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    name: text('name').notNull(),          // "ISO/IEC 42001", "ISO/IEC 23894"
    description: text('description'),       // Brief description of the standard
    createdAt: timestamp('created_at').defaultNow().notNull(),
  },
  (table) => ({
    uniqueName: unique('compliance_frameworks_name_unique').on(table.name),
  })
)
```

**Design Decisions:**
- **Unique name constraint:** Prevents duplicate framework entries
- **Simple structure:** Name and description only; versioning is handled by `framework_versions`

**Feature Usage:**
- Seed framework: `INSERT INTO compliance_frameworks (name, description) VALUES ('ISO/IEC 42001', 'AI Management System')` -- Seeded at deployment
- List frameworks: `SELECT * FROM compliance_frameworks ORDER BY name`

---

### 12. framework_versions

**Purpose:** Versioned releases of each compliance framework. Supports tracking active vs deprecated versions.

**Drizzle Schema:**
```typescript
export const frameworkVersions = pgTable(
  'framework_versions',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    frameworkId: uuid('framework_id').notNull()
      .references(() => complianceFrameworks.id, { onDelete: 'cascade' }),
    versionLabel: text('version_label').notNull(),   // "2023", "202x"
    status: text('status').notNull().$type<'active' | 'deprecated'>().default('active'),
    publishedAt: timestamp('published_at'),           // When ISO published this version
    createdAt: timestamp('created_at').defaultNow().notNull(),
  },
  (table) => ({
    frameworkIdx: index('framework_versions_framework_idx').on(table.frameworkId),
    uniqueFrameworkVersion: unique('framework_versions_framework_version_unique').on(
      table.frameworkId, table.versionLabel
    ),
  })
)
```

**Design Decisions:**
- **Version label as text:** Flexible enough for "2023", "2024-draft", etc.
- **Status field:** Allows deprecating old versions without deleting data
- **Unique (framework_id, version_label):** One version label per framework

**Feature Usage:**
- Seed version: `INSERT INTO framework_versions (framework_id, version_label, status) VALUES (...)`
- Get active version: `SELECT * FROM framework_versions WHERE framework_id = $1 AND status = 'active'`

### 13. framework_controls

**Purpose:** Individual clauses/controls within a framework version. Represents the specific requirements (e.g., "A.6.2.6 - Data quality management for AI systems").

**Drizzle Schema:**
```typescript
export const frameworkControls = pgTable(
  'framework_controls',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    versionId: uuid('version_id').notNull()
      .references(() => frameworkVersions.id, { onDelete: 'cascade' }),
    clauseRef: text('clause_ref').notNull(),     // "A.6.2.6", "6.3", etc.
    domain: text('domain').notNull(),             // "Data management", "Risk management"
    title: text('title').notNull(),               // "Data quality management for AI systems"
    createdAt: timestamp('created_at').defaultNow().notNull(),
  },
  (table) => ({
    versionClauseIdx: index('framework_controls_version_clause_idx').on(table.versionId, table.clauseRef),
    uniqueVersionClause: unique('framework_controls_version_clause_unique').on(table.versionId, table.clauseRef),
  })
)
```

**Design Decisions:**
- **Clause reference as text:** Accommodates varied numbering schemes ("A.6.2.6", "6.3", "B.1")
- **Domain grouping:** Organizes controls by subject area for reporting
- **Unique (version_id, clause_ref):** One control per clause reference per version

**Feature Usage:**
- Seed controls: `INSERT INTO framework_controls (version_id, clause_ref, domain, title) VALUES (...)` -- Bulk insert during seeding
- List controls by domain: `SELECT * FROM framework_controls WHERE version_id = $1 ORDER BY domain, clause_ref`

### 14. interpretive_criteria

**Purpose:** Guardian's own interpretive language for each framework control. Provides the assessment guidance Claude uses when evaluating vendor responses against ISO requirements.

**Drizzle Schema:**
```typescript
export const interpretiveCriteria = pgTable(
  'interpretive_criteria',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    controlId: uuid('control_id').notNull()
      .references(() => frameworkControls.id, { onDelete: 'cascade' }),
    criteriaVersion: text('criteria_version').notNull(),  // "guardian-iso42001-v1.0"
    criteriaText: text('criteria_text').notNull(),          // Guardian's interpretive language
    assessmentGuidance: text('assessment_guidance'),        // How to assess this criterion
    reviewStatus: text('review_status').notNull()
      .$type<'draft' | 'approved' | 'deprecated'>()
      .default('draft'),
    approvedAt: timestamp('approved_at'),
    approvedBy: text('approved_by'),                        // Who approved (user ID or name)
    createdAt: timestamp('created_at').defaultNow().notNull(),
  },
  (table) => ({
    controlCriteriaIdx: index('interpretive_criteria_control_version_idx').on(table.controlId, table.criteriaVersion),
    uniqueControlVersion: unique('interpretive_criteria_control_version_unique').on(table.controlId, table.criteriaVersion),
  })
)
```

**Design Decisions:**
- **Versioned criteria:** Multiple criteria versions per control (allows iteration without losing history)
- **Review workflow:** draft > approved > deprecated lifecycle
- **Unique (control_id, criteria_version):** One criteria text per version per control
- **Assessment guidance separate from criteria text:** Guidance tells Claude how to evaluate; criteria text is the actual standard interpretation

**Feature Usage:**
- Seed criteria: `INSERT INTO interpretive_criteria (control_id, criteria_version, criteria_text, ...) VALUES (...)`
- Get approved criteria: `SELECT * FROM interpretive_criteria WHERE control_id = $1 AND review_status = 'approved'`

### 15. dimension_control_mappings

**Purpose:** Many-to-many mapping between framework controls and Guardian's 10 risk dimensions. Each mapping has a relevance weight indicating how strongly a control relates to a dimension.

**Drizzle Schema:**
```typescript
export const dimensionControlMappings = pgTable(
  'dimension_control_mappings',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    controlId: uuid('control_id').notNull()
      .references(() => frameworkControls.id, { onDelete: 'cascade' }),
    dimension: text('dimension').notNull(),          // "regulatory_compliance", "privacy_risk", etc.
    relevanceWeight: real('relevance_weight').notNull().default(1.0),  // How relevant (0.0-1.0)
    createdAt: timestamp('created_at').defaultNow().notNull(),
  },
  (table) => ({
    dimensionControlIdx: index('dimension_control_mappings_dimension_idx').on(table.dimension, table.controlId),
    uniqueControlDimension: unique('dimension_control_mappings_unique').on(table.controlId, table.dimension),
  })
)
```

**Design Decisions:**
- **Relevance weight:** A control might be strongly relevant (1.0) to privacy but weakly relevant (0.3) to financial risk
- **Unique (control_id, dimension):** One mapping per control-dimension pair
- **Index on (dimension, control_id):** Fast lookup of all controls for a given dimension

**Feature Usage:**
- Seed mapping: `INSERT INTO dimension_control_mappings (control_id, dimension, relevance_weight) VALUES (...)`
- Get controls for dimension: `SELECT * FROM dimension_control_mappings WHERE dimension = $1`
- Get dimensions for control: `SELECT * FROM dimension_control_mappings WHERE control_id = $1`

### 16. assessment_compliance_results

**Purpose:** Per-assessment, per-control ISO compliance evaluation results. Stores the finding and evidence for each control evaluated during a vendor assessment.

**Drizzle Schema:**
```typescript
export const assessmentComplianceResults = pgTable(
  'assessment_compliance_results',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    assessmentId: uuid('assessment_id').notNull()
      .references(() => assessments.id, { onDelete: 'cascade' }),
    frameworkVersionId: uuid('framework_version_id').notNull()
      .references(() => frameworkVersions.id, { onDelete: 'cascade' }),
    criteriaVersion: text('criteria_version').notNull(),  // "guardian-iso42001-v1.0"
    controlId: uuid('control_id').notNull()
      .references(() => frameworkControls.id, { onDelete: 'cascade' }),
    finding: jsonb('finding'),                         // Per-control evaluation data
    evidenceRefs: jsonb('evidence_refs'),               // Q&A references supporting the finding
    createdAt: timestamp('created_at').defaultNow().notNull(),
  },
  (table) => ({
    assessmentIdx: index('assessment_compliance_results_assessment_idx').on(table.assessmentId),
    uniqueAssessmentControlVersion: unique('assessment_compliance_results_unique').on(
      table.assessmentId, table.controlId, table.criteriaVersion
    ),
  })
)
```

**Design Decisions:**
- **Triple FK (assessment, framework_version, control):** Full traceability to which version of which framework was evaluated
- **Criteria version stored:** Tracks which version of Guardian's interpretive criteria was used
- **Unique (assessment_id, control_id, criteria_version):** One finding per control per criteria version per assessment
- **JSONB for finding and evidenceRefs:** Flexible structure for different types of evaluation results

**Feature Usage:**
- Store compliance result: `INSERT INTO assessment_compliance_results (assessment_id, framework_version_id, ...) VALUES (...)`
- Get all results for assessment: `SELECT * FROM assessment_compliance_results WHERE assessment_id = $1`
- Get results by framework: `SELECT * FROM assessment_compliance_results WHERE assessment_id = $1 AND framework_version_id = $2`

## Relationships

```
users
  |-- (created_by) --> assessments
  |-- (user_id) --> conversations
  +-- (user_id) --> files

vendors
  +-- (vendor_id) --> assessments
        |-- (assessment_id) --> questions
        |-- (assessment_id) --> responses
        |-- (assessment_id) --> dimension_scores
        |-- (assessment_id) --> assessment_results
        +-- (assessment_id) --> assessment_compliance_results

conversations
  |-- (user_id) --> users
  |-- (assessment_id) --> assessments (nullable)
  |-- (conversation_id) --> messages
  +-- (conversation_id) --> files

files
  +-- (file_id) --> responses (optional source link)

compliance_frameworks
  +-- (framework_id) --> framework_versions
        +-- (version_id) --> framework_controls
              |-- (control_id) --> interpretive_criteria
              |-- (control_id) --> dimension_control_mappings
              +-- (control_id) --> assessment_compliance_results

framework_versions
  +-- (framework_version_id) --> assessment_compliance_results
```

**Foreign Key Cascade Rules:**

| Parent | Child | ON DELETE | Rationale |
|--------|-------|-----------|-----------|
| vendors | assessments | CASCADE | If vendor deleted, assessments go with it |
| assessments | questions | CASCADE | Questions belong to assessment |
| assessments | responses | CASCADE | Responses belong to assessment |
| assessments | dimension_scores | CASCADE | Scores belong to assessment |
| assessments | assessment_results | CASCADE | Results belong to assessment |
| assessments | assessment_compliance_results | CASCADE | Compliance results belong to assessment |
| conversations | messages | CASCADE | Messages belong to conversation |
| conversations | files | CASCADE | Files belong to conversation |
| users | assessments | RESTRICT (default) | Can't delete user who created assessments |
| users | conversations | RESTRICT (default) | Can't delete user who has conversations |
| users | files | CASCADE | If user deleted, their files go with them |
| files | responses | SET NULL (default) | Responses can exist without source file |
| compliance_frameworks | framework_versions | CASCADE | Versions belong to framework |
| framework_versions | framework_controls | CASCADE | Controls belong to version |
| framework_versions | assessment_compliance_results | CASCADE | Results reference version |
| framework_controls | interpretive_criteria | CASCADE | Criteria belong to control |
| framework_controls | dimension_control_mappings | CASCADE | Mappings belong to control |
| framework_controls | assessment_compliance_results | CASCADE | Results reference control |

## Feature Map

### What Each Table Supports

**Chat Foundation:**
- `users`: Who's chatting
- `conversations`: Chat sessions with mode switching and auto-titles
- `messages`: Chat history, context for resuming

**Questionnaire Generation:**
- `vendors`: Who we're assessing
- `assessments`: Track assessment lifecycle from draft to scored
- `questions`: The 87 questions Claude generated for this vendor
- `files`: Uploaded documents for intake parsing

**Risk Scoring (Epic 15/18/20/40):**
- `responses`: Extracted vendor answers from uploaded questionnaires
- `dimension_scores`: Per-dimension risk scores (10 weighted dimensions, sub-scores in JSONB)
- `assessment_results`: Composite scores (weighted average), recommendations, narrative reports, rubric v1.1 provenance

**ISO Compliance (Epic 37):**
- `compliance_frameworks`: ISO standard definitions (e.g., ISO/IEC 42001)
- `framework_versions`: Versioned releases of each standard
- `framework_controls`: Individual clauses/requirements
- `interpretive_criteria`: Guardian's interpretive assessment language per control
- `dimension_control_mappings`: Links controls to risk dimensions with relevance weights
- `assessment_compliance_results`: Per-assessment compliance findings per control

**What's NOT Yet Implemented:**
- Detailed audit log (planned for Phase 3)
- Portfolio analytics

## Performance Considerations

**Queries that need to be fast:**
1. Load conversation history (last 50 messages) - Indexed on conversation_id
2. Get vendor's assessments - Indexed on vendor_id
3. Load questions for assessment - Indexed on assessment_id
4. User's active conversations - Indexed on user_id
5. Get responses for scoring - Indexed on assessment_id + batch_id
6. Get dimension scores - Indexed on assessment_id + dimension
7. Claim narrative generation - Indexed on narrative_status
8. Get controls for a framework version - Indexed on version_id + clause_ref
9. Get compliance results for assessment - Indexed on assessment_id
10. Get dimension-control mappings - Indexed on dimension + control_id

**All queries hit indexes.** No optimization needed at current scale.
