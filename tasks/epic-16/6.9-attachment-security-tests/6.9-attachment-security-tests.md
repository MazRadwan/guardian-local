# Epic 16.6.9: Attachment Security & Test Coverage

## Overview

Code review identified gaps in Epic 16.6.8 implementation:
1. Download endpoint is user-scoped, not conversation-scoped (spec requires conversation ownership)
2. Server doesn't validate attachment payloads against owned files
3. Missing backend test coverage for attachment flows

This sprint addresses all blocking issues.

---

## Current State (Gap Analysis)

### Download Security
```
Current:
  Client sends: base64(storagePath) + filename
  Server validates: storagePath.startsWith(`uploads/${userId}/`)
  Problem: User can download ANY of their files, not just files attached to current conversation

Required (per Story 6.8.4 spec):
  Client sends: fileId
  Server validates: fileId exists AND belongs to a conversation user owns
  Server resolves: storagePath internally, never exposed to client
```

### Attachment Validation
```
Current:
  Client sends: { attachments: [{ fileId, filename, storagePath, ... }] }
  Server: Trusts client-supplied data, stores as-is

Required:
  Client sends: { attachments: [{ fileId }] }
  Server: Looks up fileId, validates ownership, enriches with server-side metadata
```

## Schema Design

### New Table: `files`

Tracks uploaded files with conversation linkage for authorization.

```typescript
// packages/backend/src/infrastructure/database/schema/files.ts
import { pgTable, uuid, text, integer, timestamp } from 'drizzle-orm/pg-core';
import { users } from './users.js';
import { conversations } from './conversations.js';

export const files = pgTable('files', {
  id: uuid('id').primaryKey().defaultRandom(),

  // Ownership
  userId: uuid('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
  conversationId: uuid('conversation_id').notNull().references(() => conversations.id, { onDelete: 'cascade' }),

  // File metadata
  filename: text('filename').notNull(),
  mimeType: text('mime_type').notNull(),
  size: integer('size').notNull(),
  storagePath: text('storage_path').notNull(), // Internal only, never sent to client

  // Timestamps
  createdAt: timestamp('created_at').defaultNow().notNull(),
});
```

### Story 6.9.5: Server-Side Attachment Validation

**Files:**
- `packages/backend/src/infrastructure/websocket/ChatServer.ts`

**Changes:**
- When `send_message` includes attachments, validate each fileId:
  1. File exists in `files` table
  2. File belongs to the authenticated user
  3. File belongs to the target conversation
- Reject message if any attachment fails validation
- Enrich attachment data from database (don't trust client-supplied metadata)

**Validation Flow:**
```typescript
// Client sends (minimal):
{ attachments: [{ fileId: 'uuid-123' }] }

// Server validates & enriches:
for (const att of attachments) {
  const file = await fileRepo.findByIdAndConversation(att.fileId, conversationId);
  if (!file) {
    socket.emit('error', { message: 'Invalid attachment' });
    return;
  }
  // Use server-side metadata
  enrichedAttachments.push({
    fileId: file.id,
    filename: file.filename,
    mimeType: file.mimeType,
    size: file.size,
    // storagePath intentionally omitted from message storage
  });
}
```

**Acceptance Criteria:**
- [ ] Attachments validated against files table
- [ ] Invalid fileId rejected with error
- [ ] File not owned by user rejected
- [ ] File not in target conversation rejected
- [ ] Attachment metadata enriched from database
- [ ] storagePath NOT stored in message attachments
