# Epic 16.6.1: Document Upload UX Improvements - Implementation Plan

## Overview

This sprint improves the document upload UX to align with industry standards (Claude.ai, ChatGPT). The key changes are:
1. **Silent context caching** - Remove visible assistant message, store context in conversation.context
2. **Inject context into Claude** - Ensure Claude sees stored context without visible message
3. **Compact file chip** - Replace full-width progress card with compact dark chip inside composer
4. **Always-visible X button** - Allow cancel/remove at any stage with proper race condition handling

---

## Architecture Summary

### Current Flow (to be modified)
```
User selects file
  → useFileUpload calls POST /api/documents/upload
  → Backend validates, stores file, parses document
  → Backend emits: upload_progress (WS)
  → Backend saves assistant message with context ❌ REMOVE
  → Backend emits: 'message' event (WS) ❌ REMOVE
  → Backend emits: intake_context_ready (WS)
  → Frontend shows UploadProgress card above composer ❌ CHANGE
```

### Target Flow
```
User selects file
  → useFileUpload calls POST /api/documents/upload
  → Backend validates, stores file, parses document
  → Backend emits: upload_progress (WS)
  → Backend stores context in conversation.context (using existing updateContext) ✅
  → Backend emits: intake_context_ready (WS)
  → Frontend shows FileChip inside composer ✅ NEW
  → On next message, buildConversationContext injects stored context ✅ NEW
```

### Story 6.1.2: Backend - Inject Stored Context into Claude Request

**Goal:** Ensure Claude sees the uploaded document context even though it's no longer a visible assistant message. This prevents functional regression.

**Why This Is Critical:**
- Currently, `buildConversationContext()` only sends user/assistant message history to Claude
- Without the assistant message, Claude will NOT know about the uploaded document
- We must inject stored `conversation.context.intakeContext` into Claude's context

**Files to Modify:**
- `packages/backend/src/infrastructure/websocket/ChatServer.ts`

**Approach: Synthetic Assistant Message (Preferred)**

Use the synthetic assistant message approach instead of system prompt augmentation to:
- Keep the base system prompt stable for prompt caching
- Avoid prompt bloat
- Maintain clear separation between system instructions and document context

   ```typescript
    * Format stored intake context as synthetic assistant message for Claude
    * This is NOT displayed to users - it's injected into Claude's message history

   private formatIntakeContextForClaude(
     ctx: IntakeDocumentContext,
     gapCategories?: string[]
   ): string {
     const parts: string[] = [
       'I have analyzed the uploaded document and extracted the following context:',
     ];

     if (ctx.vendorName) parts.push(`- Vendor: ${ctx.vendorName}`);
     if (ctx.solutionName) parts.push(`- Solution: ${ctx.solutionName}`);
     if (ctx.solutionType) parts.push(`- Type: ${ctx.solutionType}`);
     if (ctx.features?.length) parts.push(`- Key Features: ${ctx.features.slice(0, 5).join(', ')}`);
     if (ctx.complianceMentions?.length) parts.push(`- Compliance Mentions: ${ctx.complianceMentions.join(', ')}`);
     if (gapCategories?.length) parts.push(`- Areas Needing Clarification: ${gapCategories.join(', ')}`);

     parts.push('', 'I will use this context to assist with the assessment.');
     return parts.join('\n');
   }
   ```

```tsx
      {/* Error message */}
      {isError && error && (
        <span className="text-xs text-red-300 truncate">{error}</span>
      )}
    </div>
  );
}
```

### Story 6.1.5: Frontend - Handle Cancel With Race Condition Protection

**Goal:** Allow users to cancel an in-progress upload. Prevent race condition where WS events resurrect the chip after cancel.

**The Race Condition Scenarios:**

**Scenario 1: Cancel during HTTP flight**
```
1. User starts upload → HTTP in flight, uploadId unknown
2. User cancels
3. HTTP response arrives with uploadId
4. WS events for that uploadId start arriving
5. BUG: Chip resurrects!
```

**Scenario 2: New upload started, old events arrive**
```
1. Upload A completes with uploadId="A-server"
2. User starts Upload B → HTTP in flight, uploadId unknown
3. Late WS event for "A-server" arrives
4. BUG: Event adopted for Upload B (wrong upload)!
```

**Files to Modify:**
- `apps/web/src/hooks/useFileUpload.ts`

**Solution: Never Adopt from WS Events**

The key invariant: **Only process WS events for the uploadId returned by the most recent successful HTTP response.**

This eliminates the "adopt first event" pattern that creates vulnerabilities.

**CRITICAL: Apply to ALL event handlers**

The "never adopt" gating must be applied to **all three** WebSocket event handlers:
- `upload_progress` - Progress updates
- `intake_context_ready` - Parsing complete (intake mode)
- `scoring_parse_ready` - Parsing complete (scoring mode)

If any handler lacks this gating, the chip can resurrect via that event path.

**Changes:**

1. Add AbortController ref:
   ```typescript
   const abortControllerRef = useRef<AbortController | null>(null);
   ```

2. Simplify event handler logic - **never adopt** (apply to ALL handlers):
   ```typescript
   // upload_progress handler
   const unsubProgress = wsAdapter.subscribeUploadProgress((data) => {
     if (data.conversationId !== conversationId) return;

     // CRITICAL: Only accept events for known uploadId
     // Never adopt - prevents race conditions
     if (!currentUploadIdRef.current) return;
     if (data.uploadId !== currentUploadIdRef.current) return;

     // ... rest of handler (unchanged) ...
   });

   // intake_context_ready handler - SAME GATING
   const unsubIntake = wsAdapter.subscribeIntakeContextReady((data) => {
     if (data.conversationId !== conversationId) return;

     // CRITICAL: Same gating as upload_progress
     if (!currentUploadIdRef.current) return;
     if (data.uploadId !== currentUploadIdRef.current) return;

     // ... rest of handler (unchanged) ...
   });

   // scoring_parse_ready handler - SAME GATING
   const unsubScoring = wsAdapter.subscribeScoringParseReady((data) => {
     if (data.conversationId !== conversationId) return;

     // CRITICAL: Same gating as upload_progress
     if (!currentUploadIdRef.current) return;
     if (data.uploadId !== currentUploadIdRef.current) return;

     // ... rest of handler (unchanged) ...
   });
   ```

3. In `uploadFile()`, create AbortController and handle abort:
   ```typescript
   const uploadFile = useCallback(async (file: File) => {
     // ... validation code ...

     // Clear any previous upload state
     currentUploadIdRef.current = null;

     // Create AbortController for cancellation
     abortControllerRef.current = new AbortController();

     setUploadProgress({
       uploadId: null,
       progress: 10,
       stage: 'uploading',
       message: 'Uploading file...',
     });

     try {
       const response = await fetch(`${API_BASE_URL}/api/documents/upload`, {
         method: 'POST',
         headers: { 'Authorization': `Bearer ${token}` },
         body: formData,
         signal: abortControllerRef.current.signal,
       });

       if (!response.ok) {
         const errorData = await response.json().catch(() => ({ error: 'Upload failed' }));
         throw new Error(errorData.error || `Upload failed: ${response.status}`);
       }

       // Check if aborted during response parsing
       if (abortControllerRef.current?.signal.aborted) {
         return; // Cancelled, don't set uploadId
       }

       const { uploadId } = await response.json();

       // Set uploadId - now WS events for this ID will be accepted
       currentUploadIdRef.current = uploadId;

       setUploadProgress({
         uploadId,
         progress: 30,
         stage: 'storing',
         message: 'Processing upload...',
       });

     } catch (error) {
       if (error instanceof Error && error.name === 'AbortError') {
         // User cancelled - reset quietly, don't set error state
         setUploadProgress({
           uploadId: null,
           progress: 0,
           stage: 'idle',
           message: '',
         });
         return;
       }
       // ... existing error handling ...
     }
   }, [token, conversationId, mode, validateFile]);
   ```

4. In `reset()`, abort and clear uploadId:

   ```typescript
     abortControllerRef.current?.abort();
   ```

**Why This Works:**

| Scenario | Before Fix | After Fix |
|----------|------------|-----------|
| Cancel during HTTP | uploadId=null, WS adopted | uploadId=null, WS ignored |
| Cancel after HTTP | uploadId cleared, WS adopted | uploadId=null, WS ignored |
| Old event during new upload | uploadId=null, WS adopted | uploadId=null, WS ignored |
| Normal operation | Works | Works (events match uploadId) |

**Tradeoff:** If WS events arrive before HTTP response, they're ignored. This means:
- Progress might "jump" from 10% (uploading) to 30%+ (first accepted WS event)
- This is acceptable - HTTP is typically fast, and data integrity > smooth animation

## Edge Cases

### Mid-flight conversation deletion
If conversation is deleted while upload is processing:
- Backend: `updateContext()` throws "Conversation not found"
- Error propagates to `processUpload()` catch block
- Error event emitted to client
- Frontend: Shows error state in FileChip
- **Expected behavior:** No crash, graceful error display

### WS events arrive before HTTP response
- Events are ignored (uploadId not yet set)
- Progress may "jump" when HTTP completes and subsequent events arrive
- **Acceptable tradeoff:** Data integrity over smooth animation
