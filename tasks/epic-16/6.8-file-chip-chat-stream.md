# Epic 16.6.8: File Chip Chat Stream Integration

## Overview

After Sprint 16.6.1, the FileChip displays inside the composer during upload/processing. However, once the user sends a message, the chip **remains stuck in the composer** instead of moving to the chat stream as part of the user's message.

This story aligns Guardian's file upload UX with industry standards (ChatGPT, Claude.ai) where uploaded files appear as message attachments in the conversation history.

## Requirements

### 1. File Chip Moves to Chat Stream on Send

When user clicks Send (with or without text message):
- FileChip state is cleared from composer
- File attachment metadata is included with the user message
- MessageBubble renders the file attachment as part of the user message

### 2. Light Theme for FileChip

Restyle FileChip to match app's light theme:

| Property | Current (Dark) | Target (Light) |
|----------|----------------|----------------|
| Background | `bg-gray-800` | `bg-gray-100` or `bg-white border` |
| Text | `text-white` | `text-gray-900` |
| Icon | White icons | `text-blue-500` or `text-gray-600` |
| X button | `text-gray-400` | `text-gray-400 hover:text-gray-600` |
| Error bg | `bg-red-900` | `bg-red-50 border-red-200` |

### 3. File Chip in Chat Stream (MessageBubble)

Create a chat-stream variant of FileChip for rendering in MessageBubble:
- Displayed inside user message bubble (right-aligned)
- No X button (file is already sent, can't remove)
- No progress bar (already complete)
- **Clickable** - triggers file download

### 4. Download Functionality

When file chip in chat stream is clicked:
- Fetch file from backend (existing `/api/documents/:id/download` or similar)
- Trigger browser download with original filename

### 5. Not Sticky

File attachment message flows with conversation:
- New assistant responses appear below it
- User can scroll up to see previous file attachments
- No special pinning/sticky behavior

## Technical Implementation

### Data Flow

```
Current:
  Upload completes → FileChip shows "Ready" in composer → stays forever

Target:
  Upload completes → FileChip shows "Ready" in composer
  User sends message → Message includes { attachments: [{ fileId, filename }] }
                    → FileChip cleared from composer
                    → MessageBubble renders with FileAttachment component
```

### Schema Changes

**Message entity** - add optional attachments field:
```typescript
// packages/backend/src/domain/entities/Message.ts
export interface MessageAttachment {
  fileId: string;
  filename: string;
  mimeType: string;
  size: number;
  storagePath: string;
}

export interface Message {
  // ... existing fields
  attachments?: MessageAttachment[];
}
```

**Database schema** - add attachments column:
```typescript
// packages/backend/src/infrastructure/database/schema/messages.ts
attachments: jsonb('attachments').$type<MessageAttachment[]>(),
```

### Backend Changes

1. **Message creation** - Accept attachments in send message payload
2. **File download endpoint** - Serve stored files (may already exist)

### Frontend Changes

1. **Composer.tsx** - Pass attachment info when sending message, clear chip after send
2. **useFileUpload.ts** - Expose `getAttachment()` method to get file metadata for message
3. **FileChip.tsx** - Restyle to light theme
4. **FileChipInChat.tsx** - New component for chat stream (no X, clickable)
5. **MessageBubble.tsx** - Render file attachments from message data

### Story 6.8.8: Frontend - Download Handler

**Files:**
- `apps/web/src/hooks/useFileDownload.ts` (NEW) or inline in ChatInterface

**Implementation:**
```typescript
const downloadFile = async (fileId: string, filename: string) => {
  const response = await fetch(`/api/documents/${fileId}/download`, {
    headers: { Authorization: `Bearer ${token}` },
  });
  const blob = await response.blob();
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = filename;
  a.click();
  URL.revokeObjectURL(url);
};
```

**Acceptance Criteria:**
- [ ] Clicking file in chat triggers download
- [ ] File downloads with original filename
- [ ] Auth token included in request
- [ ] Error handling for failed downloads

## Notes

1. **MVP Scope:** This story focuses on the core UX pattern. Advanced features like drag-and-drop, multi-file, or inline preview are out of scope.

2. **Backward Compatibility:** Existing messages without attachments are unaffected (nullable field).

3. **File Storage:** Uses existing file storage from Epic 16. No changes to how files are stored.

4. **Security:** Download endpoint must verify user owns the conversation containing the file attachment.
