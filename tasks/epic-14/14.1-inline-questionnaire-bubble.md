## Story 14.1 – Inline Questionnaire Chat Bubble

Status: Complete

### Overview
Render the questionnaire generation/ready UI as a normal assistant chat message (with avatar, timestamp, and standard padding) instead of a special floating card. The bubble must appear in the message history where it was produced, not pinned at the bottom or behind the composer.

  - `apps/web/src/components/chat/ChatMessage.tsx`

  - `apps/web/src/components/chat/ChatInterface.tsx`

  - `apps/web/src/stores/chatStore.ts` (data structure for questionnaire message insertion)

### Requirements
- Questionnaire UI renders as a standard assistant message: avatar, role label “Guardian”, normal chat padding.
- Bubble appears at the point in the transcript where generation started; new chat messages append after it (no sticky/floating behavior).
- Generation/loading/complete states stay visible; stepper content remains but lives inside the chat message layout.
- Composer should not occlude the bubble; scroll behavior should keep message visible when near bottom without forcing stickiness.
- Retain current logic for generation state, stepper, persistence, and download flow.

### Design Decisions
| Decision | Choice | Rationale |
|----------|--------|-----------|
| Placement | Convert questionnaire UI to a chat message entry | Keeps chronological order and consistent UX |
| Styling | Reuse chat message layout; no new card container | Aligns with existing assistant bubbles |
| Slot removal | Remove/avoid `questionnaireSlot` rendering | Eliminates sticky-bottom behavior |
| State | Reuse existing store payloads/UI state | Avoids regression in persistence and phase handling |

### Testing
- Component tests: ChatMessage (renders questionnaire bubble), MessageList ordering, ChatInterface layout.
- Interaction: generation start, progression, completion, subsequent user message renders after the questionnaire bubble.
- Visual: snapshot/DOM assertions for avatar + standard padding; no overlap with composer.
