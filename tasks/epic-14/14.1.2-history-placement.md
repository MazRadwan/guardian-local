## Story 14.1.2 – Place Questionnaire Bubble in History

Status: Complete

### Goal
Insert the questionnaire bubble into the chat transcript at the point of generation instead of rendering it as a sticky slot. New messages should append after it in chronological order.

### Prerequisites
- **Story 14.1.1 MUST be complete** (QuestionnaireMessage component exists)
- Read: `apps/web/src/components/chat/ChatInterface.tsx`
- Read: `apps/web/src/components/chat/MessageList.tsx`
- Read: `apps/web/src/stores/chatStore.ts`

---

## Current State Analysis

### How It Works Now

1. **ChatInterface.tsx (lines 256-282):**
   - Passes `questionnaireSlot` prop to `MessageList`
   - Conditionally renders `QuestionnairePromptCard` inside the slot
   - Slot is rendered AFTER all messages in MessageList

2. **MessageList.tsx (line 139):**
   - Renders `{questionnaireSlot}` after the messages map
   - This makes the questionnaire appear at the end, not inline with history

3. **chatStore.ts:**
   - `pendingQuestionnaire` stores the payload separately from messages
   - Not part of the `messages[]` array

### Problem

The questionnaire renders AFTER all messages because it's a separate slot, not a message in the array. This means:
- It always appears at the bottom (sticky behavior)
- New messages don't push it up
- It doesn't have a position in the conversation history

---

## Implementation Approach

### Strategy: Virtual Message Injection

Instead of modifying the core messages array (which would require backend changes), we'll:
1. Compute the questionnaire's position based on when `pendingQuestionnaire` was set
2. Inject it into the rendered message list at that position
3. Remove the `questionnaireSlot` prop entirely

This approach:
- Preserves existing persistence logic
- Doesn't require backend changes
- Works with existing rehydration flow

## Edge Cases & Considerations

### Streaming Edge Case

**Scenario:** What if a streaming assistant message is active when the questionnaire appears?

**Behavior:** The `questionnaire_ready` event is emitted by Claude AFTER the assistant finishes its message explaining what will be generated. By design, the questionnaire should appear after that message.

**Implementation Guard:** The index is captured at `setPendingQuestionnaire` time, which happens when `questionnaire_ready` is processed - AFTER the assistant message is complete and added to messages array.

**If streaming IS active when questionnaire_ready arrives (edge case):**
- The index will be set to current `messages.length`
- This places questionnaire after the last COMPLETE message
- The streaming message will continue appending to itself
- On stream complete, the message is finalized
- Questionnaire remains at its captured position

**Test this scenario:** Verify questionnaire doesn't interrupt mid-stream content.

### Rehydration Handling

**Scenario:** User refreshes page or returns to a conversation with an existing questionnaire.

**Current Flow (via useQuestionnairePersistence hook):**
1. On conversation load, hook checks for persisted payload
2. If found, calls `setPendingQuestionnaire(payload)` and `setQuestionnaireUIState(state)`
3. This triggers index capture based on CURRENT messages.length

**Potential Issue:** On rehydration, messages are loaded first, THEN questionnaire state is restored. The index would be set to `messages.length` (end of history), which is correct for rehydration - the questionnaire appears at the end.

**If questionnaire should appear at original position:**
We would need to persist the index alongside the payload. However, this adds complexity and the messages array might differ on reload (pagination, etc.).

**Recommendation:** Keep current behavior - on rehydration, questionnaire appears at end. This is acceptable because:
- User expects to see it when returning
- Original conversation position is less important after session break
- Simpler, less error-prone

### Duplicate Prevention

**Scenario:** Rapid conversation switches could create multiple questionnaire entries.

**Guards in place:**
1. `setActiveConversation` calls `clearPendingQuestionnaire()` - clears on switch
2. ChatInterface guard: `pendingQuestionnaire.conversationId === activeConversationId`
3. Index reset to -1 on clear

**Test:** Switch conversations rapidly and verify no duplicates render.
