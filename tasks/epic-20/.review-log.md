**Must-Change Items Raised:**

1. **Clean Architecture**: `ScoringExportService` should not directly build prompts or talk to vendor clients; introduce an application-layer port (e.g., `INarrativeGenerator`/`IExportNarrativeService`) with an infrastructure implementation.

3. **Security/PII**: Cleanup job audit logs must log counts/IDs only (no response content). Prompt caching must be static prompt only (never user responses).

4. **Parallelization Reality**: Sprint 3/4 items touch shared files. Split into "interface changes first" then "call-site implementations".

**Recommendations:**
- Add unit tests for `determineSolutionType()`, fallback, transactions, abort
- Unit of Work pattern for transactions (don't leak Drizzle types)
- Conservative orphan definition with dry-run mode
- Token estimation instead of pure char count

### Round 3: Claim Pattern Discussion

**Options Discussed:**
1. Sentinel value (`__IN_PROGRESS__`) in narrative_report
2. Add `narrative_status` enum column
3. PostgreSQL advisory lock

**GPT Recommendation:** Option 2 - `narrative_status` column is cleanest:
- Separates data from state
- Observable + debuggable
- Crash-safe
- Works well with multiple workers

---

### Round 4: Final Approval

**Agent Proposed:**
- `narrative_status`: `null` | `'generating'` | `'complete'` | `'failed'`
- `narrative_claimed_at`: timestamp for stuck claim recovery
- Claim via atomic `UPDATE ... RETURNING`
- 5 min TTL for stuck claims

**GPT Response:** APPROVED with additional recommendations:
- Don't keep DB transaction open during LLM call
- Guard finalize update on claim timestamp
- Use `timestamptz` for claimed_at
- Consider 202 Accepted + Retry-After for concurrent requests
- Optional: CHECK constraint, narrative_error, narrative_completed_at
