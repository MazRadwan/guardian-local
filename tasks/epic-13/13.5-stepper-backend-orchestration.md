# Story 13.5 – Stepper Backend Orchestration

## Status: Not Started

## Phase Event Design

### Event Names
| Event | Phase Index | Trigger Point | Duration |
|-------|-------------|---------------|----------|
| `generation_phase` | 0 | Before Claude call | Instant |
| `generation_phase` | 1 | After Claude returns | 10-15s (Claude latency) |
| `generation_phase` | 2 | After validation | 100-500ms |
| `generation_phase` | 3 | After persistence | 100-200ms |
| `export_ready` | 4 (complete) | After markdown ready | Instant |

### Event Payload
```typescript
interface GenerationPhasePayload {
  conversationId: string;
  phase: number;
  phaseId: 'context' | 'generating' | 'validating' | 'saving';
  timestamp: number;
}
```

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Single event vs multiple | Single `generation_phase` | Simpler handler, consistent with step array |
| Emit from ChatServer vs Service | ChatServer | Already has socket reference, cleaner |
| Phase numbering | 0-3 + export_ready=4 | Matches GENERATION_STEPS array indices |
| Idempotency | Only advance forward | Handles reconnection, duplicate events |
| Start step | Frontend sets step 0 | Immediate feedback before network round-trip |

## Error Handling

| Error Point | Phase Emitted | Action |
|-------------|---------------|--------|
| Before Claude call | Phase 0 only | Error event resets to -1 |
| Claude call fails | Phase 0 only | extraction_failed resets to -1 |
| Validation fails | Phases 0-1 | extraction_failed resets to -1 |
| Persistence fails | Phases 0-2 | extraction_failed resets to -1 |

## Non-Goals

- Reconnection state persistence (in-memory phases only)
- Dynamic step labels from backend
- Changing download durability (13.3)
- UI changes beyond wiring (13.4 scaffold used as-is)
