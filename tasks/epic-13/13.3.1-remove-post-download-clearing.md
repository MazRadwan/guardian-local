# Story 13.3.1 - Remove Post-Download State Clearing

## Status: Not Started

## Context
After a successful download, the current `handleDownload` function clears all questionnaire state, hiding the download buttons. This forces users to regenerate the questionnaire if they want to download again (e.g., in a different format).

## Goal
Remove the 3 state-clearing calls from `handleDownload` so download buttons remain visible after successful download, allowing multiple downloads without regeneration.

## File to Modify
`apps/web/src/components/chat/ChatInterface.tsx`

## Current Implementation
In `handleDownload` (around lines 175-180), after successful download:

```typescript
// After successful download, clear state and persist
if (activeConversationId) {
  persistence.clearPayload(activeConversationId);
}
clearPendingQuestionnaire();
setQuestionnaireUIState('hidden');
```

## Required Changes

### Remove the 3 Clearing Calls
Delete the lines that clear state after download:

**Before:**
```typescript
try {
  // ... download logic ...

  // Create blob and download
  const blob = new Blob([response.data], { type: contentType });
  const url = window.URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = `questionnaire.${format}`;
  document.body.appendChild(a);
  a.click();
  window.URL.revokeObjectURL(url);
  document.body.removeChild(a);

  // After successful download, clear state and persist
  if (activeConversationId) {
    persistence.clearPayload(activeConversationId);
  }
  clearPendingQuestionnaire();
  setQuestionnaireUIState('hidden');
} catch (error) {
  // ... error handling (keeps UI visible for retry) ...
}
```

**After:**
```typescript
try {
  // ... download logic ...

  // Create blob and download
  const blob = new Blob([response.data], { type: contentType });
  const url = window.URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = `questionnaire.${format}`;
  document.body.appendChild(a);
  a.click();
  window.URL.revokeObjectURL(url);
  document.body.removeChild(a);

  // Keep download state visible for re-downloads
  // State only clears on: conversation delete, new questionnaire_ready, or extraction error
} catch (error) {
  // ... error handling (keeps UI visible for retry) ...
}
```

## State Lifecycle After Change

| Event | State Change | Rationale |
|-------|--------------|-----------|
| Download success | No change | Allow re-downloads |
| Download error | No change | Allow retry (existing behavior) |
| Conversation delete | Clear all | Cleanup (existing behavior) |
| New questionnaire_ready | Reset to 'ready' | New questionnaire replaces old |
| Extraction error | Clear export | Error recovery (existing behavior) |

## State Clearing Checklist (Existing Paths - No Changes Needed)
Reviewers may worry exports live forever. Document where state IS cleared:

| Location | File | Trigger | What Clears |
|----------|------|---------|-------------|
| `handleConversationDeleted` | useWebSocketEvents.ts | User deletes conversation | `clearExportReady(conversationId)` |
| `handleExtractionFailed` | useWebSocketEvents.ts | Generation fails | `clearExportReady(conversationId)` |
| `handleQuestionnaireReady` | useWebSocketEvents.ts | New questionnaire replaces old | Old payload overwritten (implicit) |
| Rehydration effect | ChatInterface.tsx | Conversation switch | `clearPendingQuestionnaire()` |

These paths remain unchanged and ensure stale exports are properly cleaned up.

## Non-Goals (Out of Scope)
- **Double-click throttling**: Preventing rapid multiple download clicks is out of scope for Story 13.3. If users click multiple times quickly, multiple fetches may occur. This can be tracked as a separate UX enhancement if needed.

## Acceptance Criteria

- [ ] User can click download multiple times (PDF, Word, Excel)
- [ ] localStorage payload NOT cleared after download
- [ ] pendingQuestionnaire NOT cleared after download
- [ ] UI state remains 'download' after download
- [ ] Error handling unchanged (UI stays visible for retry)
