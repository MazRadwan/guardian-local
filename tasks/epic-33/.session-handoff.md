## Current Issue #2: Typing Indicator Position Bug (NEW)

**Actual:**
```
[User message]
```

**Root cause hypothesis:**
- Claude streams some text first ("Let me try...")
- That text is appended as a message

2. Replace/hide partial text when tool_status changes to 'searching'

## Previous Issue: Search Stops Mid-Response

**Symptom:** Claude says "Let me try a more focused search:" with a colon, then the response stops. The message shows Copy/Regenerate buttons (meaning `assistant_done` was emitted).

**Screenshot shows:** User asked "can you do a more canada specific search" → Guardian replied "Let me try a more focused search:" → Nothing happened after

### Root Cause Hypothesis

1. **Claude didn't emit a `tool_use` block** - it only TALKED about searching instead of actually calling the tool
2. The `stopReason` was likely `end_turn` not `tool_use`
3. This is the SAME issue seen earlier in the session

### Debug Logs to Check

Backend terminal should show these logs (added during this session):
```
[MessageHandler] Claude stop_reason: ...
[MessageHandler] Claude emitted X tool_use block(s): ...
[MessageHandler] Tool loop check: mode=..., stopReason=..., toolUseBlocks=...
```

If `stopReason` is `end_turn` and `toolUseBlocks=0`, then Claude decided not to use the tool.

### Possible Causes

1. **Prompt instructions conflict** - The prompt says "DO NOT use web search for: Simple explanations of frameworks like PIPEDA" which may confuse Claude
2. **Second search limitation** - v1 design limits to ONE tool call per query. After the first search, continuation doesn't have tools available (intentional)
3. **Anthropic rate limit** - 30k input tokens/minute could cause truncation

- continueWithToolResult omits tools: `MessageHandler.ts:1007-1019` (comment explains v1 limitation)
